# Homework 2: Requirements Elicitation & Analysis

This repository contains the comprehensive **Requirements Elicitation Document** developed for **Homework 2** of the **Modern Methods of Software Engineering** course (September 2024). Building upon the foundations of Homework 1, this project focuses on the formal process of identifying, documenting, and organizing system requirements.

## Project Overview

The objective of this homework is to produce a full requirements elicitation document for a complex business system. This involves transforming a high-level business description into a structured set of technical and functional specifications.

### Key Components of the Document
1.  **Project Glossary**: A centralized dictionary defining all roles, use cases, and domain-specific concepts used throughout the documentation to ensure stakeholder alignment.
2.  **Actor Identification**: Formal definition of all human and system entities interacting with the software.
3.  **Scenario Development**: Detailed narrative descriptions of specific instances of system use (minimum 4 unique scenarios) to capture real-world operational contexts.
4.  **UML Use Case Modeling**: 
    * **Comprehensive Use Case Diagram**: A visual map of all system functionalities.
    * **Textual Descriptions**: Deep dives into major use cases (minimum 4), including initiating actors, preconditions, postconditions, "Happy Path" flows, and alternative/exception flows.
    * **Advanced Relationships**: Strategic use of *include* and *extend* relationships to manage system complexity.
5.  **Non-Functional Requirements (NFRs)**: Specification of system attributes such as usability, reliability, performance, and security.

---

## Methodologies Applied

The elicitation process followed modern software engineering standards to ensure a robust system design:

* **Elicitation Techniques**: Analyzing business descriptions to extract hidden requirements and potential edge cases.
* **Traceability**: Structuring the document so that scenarios directly inform the identified use cases, providing a clear path from user needs to system design.
* **UML Standards**: Adherence to formal UML notation for behavioral modeling.
